01 / The Problem

Find The Shadows

Two vehicles are accompanying if they appear at the same checkpoint within a 300-second window at least 10 times across 31 days.
M 5 nodes · 8 cores · 8 GB RAM each — severe memory constraints
D ~200 GB total cluster storage — cannot materialise all pairs
S Self-join on dense buckets → O(N²) candidate explosion
B Boundary effect — pairs split across time bucket edges are missed

03 / Algorithm Evolution

From Brute Force to Plane Sweep

Scala Brute Force

Scala Completed

Global collect_list + UDF combinations(2). Works on small samples; OOM on full 275M dataset due to unbounded shuffle.

ComplexityO(N²) per bucket
val pairs = grouped
  .agg(collect_list("plate"))
  .withColumn("pair",
    explode(generatePairsUDF($"plates")))
→

Iterative Chunking

Scala Completed

Processes data in 0.25h micro-batches. Overlapping buckets (+1 shift) solve boundary effect. Midpoint ownership deduplicates.

ComplexityO(N²) — bounded chunk
// Boundary fix: duplicate each record
.withColumn("bucket",
  explode(array(base, base+1)))
// Ownership: midpoint check
.filter(midpoint >= chunkStart)
→

Waterfall Aggregation

Python Completed

Hierarchical compaction: 10 Raw → 1 L1 → 10 L1 → 1 L2. Intermediate files deleted on-the-fly. Disk stays under 200 GB.

Disk PressureControlled
class WaterfallAggregator:
  def add_raw_chunk(self, df):
    if len(pending) >= l1_threshold:
      self._compact_level_1()
→

FP-Growth Mining

Python Completed

MLlib FPGrowth on (location, bucket) transactions. Dual-grid (0 + window/2 offset) handles boundaries. Finds convoy groups ≥ 2.

ScalabilitySub-quadratic
fp = FPGrowth(itemsCol="items",
  minSupport=local_min_support)
model = fp.fit(transactions)
model.freqItemsets.filter(size >= 2)
→

PandasUDF sort-then-sweep eliminates self-join entirely. MinHash LSH pre-filters stranger pairs globally. Zero data duplication.

ComplexityO(N log N + N·k)

05 / Infrastructure

Slurm → Dynamic Hadoop Cluster

Dynamic Config Injection

No hardcoded IPs. Slurm exposes $SLURM_NODELIST; the launch script parses hostnames at runtime and writes core-site.xml, hdfs-site.xml, yarn-site.xml before any daemon starts.

sbatch cluster_launch.sh
scontrol → parse node hostnames
write Hadoop XML configs dynamically
scp configs to all workers
start HDFS NameNode + DataNodes
start Spark Master + Workers
spark-submit → job runs
teardown on job exit

No Root Required

All processes (HDFS daemons, Spark Master/Worker) run as the submitting user inside Slurm's allocated container cgroups. Tmp dirs use /tmp/hadoop-$USER-$SLURM_JOB_ID to avoid conflicts between concurrent jobs.

Spark Standalone (not YARN)

Using Spark's own standalone cluster manager avoids bootstrapping a YARN ResourceManager. Workers connect directly to the elected master via spark://NODE0:7077.
